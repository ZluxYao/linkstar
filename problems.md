# LinkStar 问题汇总

## 已知问题

### 1. 长时间运行后连接断开

**问题描述**：
程序运行一段时间后（通常为数小时至数天），穿透连接会断开，导致外部无法访问内网服务。

**可能原因**：
- NAT 映射超时：大多数家庭路由器的 UDP 端口映射有超时机制，通常 300-600 秒无流量后会自动失效
- STUN 绑定失效：STUN 打洞获得的公网 IP:Port 绑定有生命周期
- 目标网络变化：公网 IP 变化导致原有映射失效

**影响范围**：
- 所有使用 STUN 穿透的服务
- 依赖 UDP 的连接

**建议修复方向**：
- 实现心跳保活机制，定期发送 STUN 请求维持端口映射
- 检测到连接断开后自动重新打洞
- 添加连接状态监控和告警

---

### 2. 高并发量时 UPnP 超时

**问题描述**：
当服务数量较多或并发请求量较大时，UPnP 端口映射操作会超时失败，导致部分或全部端口映射不成功。

**可能原因**：
- UPnP 设备响应慢：部分路由器 UPnP 性能较差，响应延迟高
- 并发竞争：多个服务同时发起 UPnP 请求，超出路由器处理能力
- UPnP 超时时间设置过短：当前超时配置无法满足大流量场景
- 路由器 UPnP 限制：部分路由器对同时存在的映射数量有限制

**影响范围**：
- 依赖 UPnP 的服务
- 多服务场景

**建议修复方向**：
- 增加 UPnP 操作超时时间
- 实现 UPnP 请求队列，避免并发竞争
- 添加 UPnP 重试机制
- 增加失败降级策略（如仅使用 STUN）

---

## 优化建议

### 短期优化
1. **心跳保活**：实现 UDP 心跳，每 60-120 秒发送一次 STUN 请求维持映射
2. **自动重连**：检测到连接断开后自动重新打洞
3. **UPnP 超时调整**：将 UPnP 超时时间从当前值调整为 5-10 秒
4. **UPnP 队列化**：使用互斥锁或队列避免并发竞争

### 长期优化
1. **连接池管理**：建立连接池，统一管理所有穿透连接
2. **状态监控**：添加连接状态监控页面
3. **故障转移**：支持配置备用穿透方案
4. **日志完善**：增加详细的连接状态和错误日志
