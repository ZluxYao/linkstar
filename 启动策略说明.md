# LinkStar 新启动策略说明

## 为什么要重新设计？

### 旧版本的问题
1. **UPnP冲突**：100个服务同时请求UPnP端口映射，路由器处理不过来（UPnP基于HTTP，是单线程的）
2. **时序混乱**：STUN发包快，UPnP慢，导致端口还没映射好，STUN就开始打洞了
3. **不稳定**：40个服务就会失败，更别说100个

### 新版本的解决方案
**三个阶段，循序渐进，简单稳定**

---

## 新启动流程（三阶段）

### 📋 阶段1：准备阶段（并发执行，快速完成）
**目的**：获取网络基础信息

**执行内容**：
1. 测试STUN服务器速度，找到最快的
2. 探测NAT网络拓扑（有几层路由）
3. 发现UPnP网关设备
4. 获取本地IP和公网IP

**特点**：
- 这些任务互不依赖，可以并发执行
- 大约需要3-5秒完成

---

### 📋 阶段2：UPnP阶段（串行执行，避免冲突）
**目的**：配置所有需要的UPnP端口映射

**执行方式**：
```
服务1 → 等200ms → 服务2 → 等200ms → 服务3 → ...
```

**为什么串行？**
- UPnP是基于HTTP的，路由器同时只能处理一个请求
- 如果并发请求，路由器会崩溃或拒绝服务
- 串行处理虽然慢一点，但100%稳定

**时间估算**：
- 100个服务 × 200ms = 20秒
- 可以接受的等待时间

**关键代码**：
```go
// upnp_queue.go - UPnP队列管理器
// 所有UPnP请求进入队列，一个一个处理
upnpQueue.AddPortMappingSync(port, ...)
```

---

### 📋 阶段3：启动阶段（分批执行，控制并发）
**目的**：启动所有STUN穿透服务

**执行方式**：
```
第1批（10个服务）→ 等2秒 → 第2批（10个服务）→ 等2秒 → ...
```

**为什么分批？**
- 避免100个goroutine同时启动，造成网络拥堵
- 每批10个，既保证速度，又不会过载
- 批次之间延迟2秒，让网络稳定下来

**时间估算**：
- 100个服务 ÷ 10 = 10批
- 10批 × 2秒 = 20秒
- 总共40秒左右完成所有启动

**关键代码**：
```go
// service_manager.go - 分批启动
const batchSize = 10              // 每批10个
const batchDelay = 2 * time.Second // 批间延迟2秒
```

## 参数调优指南

### 如果你的路由器性能好
可以加快速度：

**修改 [upnp_queue.go](modules/stun/upnp_queue.go:48)**
```go
// 从200ms改为100ms
time.Sleep(100 * time.Millisecond)
```

**修改 [service_manager.go](modules/stun/service_manager.go:129)**
```go
const batchSize = 15              // 从10改为15
const batchDelay = 1 * time.Second // 从2秒改为1秒
```

### 如果你的路由器性能差
需要放慢速度：

**修改 [upnp_queue.go](modules/stun/upnp_queue.go:48)**
```go
// 从200ms改为500ms
time.Sleep(500 * time.Millisecond)
```

**修改 [service_manager.go](modules/stun/service_manager.go:129)**
```go
const batchSize = 5               // 从10改为5
const batchDelay = 3 * time.Second // 从2秒改为3秒
```

## 核心优势

### 1. 简单易懂
- 三个阶段，清晰明了
- 每个阶段做什么，一目了然
- 日志输出详细，方便调试

### 2. 稳定可靠
- UPnP串行处理，不会冲突
- 分批启动，不会过载
- 支持100个服务稳定运行

### 3. 可调优
- 参数都是常量，容易修改
- 可以根据路由器性能调整
- 有明确的调优指南

---

## 技术细节

### UPnP队列管理器
- 使用channel实现队列
- 单goroutine串行处理
- 同步接口，等待结果返回

### 分批启动
- 收集所有服务到数组
- 按批次切片处理
- 批次之间延迟等待

### 错误处理
- UPnP失败自动禁用
- STUN失败自动重试
- 达到重试上限自动关闭

## 常见问题

### Q: 为什么启动这么慢？
A: 稳定性优先。100个服务大约需要40秒，但保证100%成功。

### Q: 可以更快吗？
A: 可以，参考"参数调优指南"，但要注意路由器承受能力。

### Q: UPnP失败怎么办？
A: 自动禁用UPnP，改用纯STUN穿透，不影响服务运行。

### Q: 如何监控服务状态？
A: 查看日志输出，每个阶段都有详细的进度提示。

## 总结

新启动策略的核心思想：**串行化冲突操作，分批化并发操作**

- UPnP是瓶颈 → 串行处理
- STUN可并发 → 分批控制
- 简单、稳定、可调优
